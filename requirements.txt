\section{Introducción}

El presente documento describe las dependencias Python necesarias para reproducir el entorno
de ejecución del proyecto \textit{Impacto Global de la IA en el Empleo}. El pipeline de datos
sigue una arquitectura Medallón (Workload → Landing → Curated → Functional) sobre un clúster
Hadoop pseudo-distribuido, con PySpark como motor de procesamiento, Hive como catálogo de
metadatos y MongoDB como destino de la capa Gold.

Las dependencias se encuentran declaradas en el archivo \texttt{requirements.txt} ubicado
en la raíz del repositorio y se instalan con:

\begin{lstlisting}
pip install -r requirements.txt
\end{lstlisting}

% ─────────────────────────────────────────────────────────────
\section{Dependencias principales}

\subsection{PySpark — \texttt{pyspark==3.5.0}}

Interfaz Python para \textbf{Apache Spark}. Es el motor central del pipeline: los cuatro
scripts de procesamiento (\texttt{workload.py}, \texttt{landing.py}, \texttt{curated.py},
\texttt{functional.py}) son todos jobs de PySpark ejecutados vía \texttt{spark-submit}
sobre YARN. Permite procesar el dataset de forma distribuida, crear tablas Hive y escribir
resultados en formatos AVRO y Parquet.

\subsection{PyMongo — \texttt{pymongo==4.6.1}}

Cliente oficial de Python para \textbf{MongoDB}. Se utiliza en los scripts
\texttt{conexion\_mongodb.py} y \texttt{migration\_data\_mongodb.py} para exportar el
resultado de la capa Functional hacia la colección \texttt{gold} de MongoDB
(\texttt{172.17.208.1:27017}), que actúa como capa de servicio para Power BI.

\subsection{PyArrow — \texttt{pyarrow==14.0.2}}

Biblioteca de serialización columnar basada en Apache Arrow. Requerida por PySpark para
leer y escribir archivos en formato \textbf{Parquet} (capas Curated y Functional).
Sin esta dependencia, las escrituras Parquet fallan en entornos locales.

\subsection{FastAVRO — \texttt{fastavro==1.9.3}}

Implementación rápida del formato \textbf{AVRO} en Python. Se utiliza para validar y
manipular el esquema \texttt{ai\_impact\_jobs.avsc} fuera del contexto de Spark, así como
para pruebas de lectura de los archivos generados en la capa Landing.

\subsection{Pandas — \texttt{pandas==2.1.4}}

Librería de manipulación de datos tabulares. Se emplea en tareas de exploración,
validación rápida de resultados y en el notebook \texttt{pyspark.ipynb} del directorio
\texttt{datalakehouse/} para comparar subconjuntos del dataset con DataFrames de Spark.

\subsection{NumPy — \texttt{numpy==1.26.4}}

Biblioteca de cómputo numérico. Dependencia transitiva de Pandas y de las operaciones
estadísticas realizadas durante el análisis exploratorio del dataset.

\subsection{Python-dotenv — \texttt{python-dotenv==1.0.1}}

Permite cargar variables de entorno desde el archivo \textbf{\texttt{.env}} en tiempo de
ejecución. El proyecto usa este mecanismo para gestionar parámetros sensibles como la URI
de conexión a MongoDB (\texttt{MONGO\_URI}), rutas HDFS y credenciales, evitando que se
expongan en el código fuente o en el repositorio público de GitHub.

\subsection{Jupyter — \texttt{jupyter==1.0.0}}

Entorno de notebooks interactivos. Requerido para ejecutar \texttt{pyspark.ipynb}
del directorio \texttt{datalakehouse/}, donde se realiza el análisis exploratorio de los
datos procesados y se documentan las consultas analíticas sobre la capa Curated.

\subsection{Loguru — \texttt{loguru==0.7.2}}

Librería de logging estructurado. Reemplaza el módulo estándar \texttt{logging} en los
scripts de migración, permitiendo registrar eventos del pipeline con niveles, colores y
formato legible sin configuración adicional.

\subsection{Pytest — \texttt{pytest==8.1.1}}

Framework de pruebas unitarias para Python. Permite ejecutar pruebas de validación sobre
las funciones auxiliares de los scripts del pipeline, asegurando la integridad del código
ante cambios.

\section{Notas de instalación}

\begin{itemize}
  \item Se recomienda usar \textbf{Python 3.10} o superior, compatible con PySpark 3.5.
  \item En entornos con Hadoop instalado, \textbf{no instalar} una versión de PySpark
        diferente a la del clúster para evitar conflictos de versión en \texttt{spark-submit}.
  \item Las variables de entorno deben definirse copiando \texttt{.env.example} a
        \texttt{.env} y completando los valores reales (URI MongoDB, rutas HDFS, etc.).
  \item Para verificar la instalación:
\begin{lstlisting}
python -c "import pyspark; print(pyspark.__version__)"
python -c "import pymongo; print(pymongo.__version__)"
\end{lstlisting}
\end{itemize}
